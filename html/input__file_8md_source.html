1 # The Input File: parameters.in {#input_file}
2 
3 Most PRISMS-PF users will spend the majority of their time interacting with the input file, parameters.h. This file allows users to specify the computational domain, the mesh, the time step parameters, the boundary conditions, the output, and model constants (and more). This file is read as a text file, so modifications to it do not require recompilation of the application.
4 
5 Here is an example of an input file from the [allenCahn application](https://github.com/prisms-center/phaseField/blob/master/applications/allenCahn/parameters.in):
6 ```
7 # Parameter list for the Allen-Cahn example application
8 # Refer to the PRISMS-PF manual for use of these parameters in the source code.
9 
10 # =================================================================================
11 # Set the number of dimensions (2 or 3 for a 2D or 3D calculation)
12 # =================================================================================
13 set Number of dimensions = 2
14 
15 # =================================================================================
16 # Set the length of the domain in all three dimensions
17 # (Domain size Z ignored in 2D)
18 # =================================================================================
19 # Each axes spans from zero to the specified length
20 set Domain size X = 100
21 set Domain size Y = 100
22 set Domain size Z = 100
23 
24 # =================================================================================
25 # Set the element parameters
26 # =================================================================================
27 # The number of elements in each direction is 2^(refineFactor) * subdivisions
28 # Subdivisions Z ignored in 2D
29 # For optimal performance, use refineFactor primarily to determine the element size
30 set Subdivisions X = 1
31 set Subdivisions Y = 1
32 set Subdivisions Z = 1
33 
34 set Refine factor = 8
35 
36 # Set the polynomial degree of the element (allowed values: 1, 2, or 3)
37 set Element degree = 1
38 
39 # =================================================================================
40 # Set the time step parameters
41 # =================================================================================
42 # The size of the time step
43 set Time step = 1.0e-2
44 
45 # The simulation ends when either the number of time steps is reached or the
46 # simulation time is reached.
47 set Number of time steps = 5000
48 
49 # =================================================================================
50 # Set the output parameters
51 # =================================================================================
52 # Type of spacing between outputs ("EQUAL_SPACING", "LOG_SPACING", "N_PER_DECADE",
53 # or "LIST")
54 set Output condition = EQUAL_SPACING
55 
56 # Number of times the program outputs the fields (total number for "EQUAL_SPACING"
57 # and "LOG_SPACING", number per decade for "N_PER_DECADE", ignored for "LIST")
58 set Number of outputs = 5
59 
60 # The number of time steps between updates being printed to the screen
61 set Skip print steps = 1000
62 
63 # =================================================================================
64 # Set the boundary conditions
65 # =================================================================================
66 # Set the boundary condition for each variable, where each variable is given by
67 # its name, as defined in equations.h. The four boundary condition
68 # types are NATURAL, DIRICHLET, NON_UNIFORM_DIRICHLET and PERIODIC. If all
69 # of the boundaries have the same boundary condition, only one boundary condition
70 # type needs to be given. If multiple boundary condition types are needed, give a
71 # comma-separated list of the types. The order is the miniumum of x, maximum of x,
72 # minimum of y, maximum of y, minimum of z, maximum of z (i.e left, right, bottom,
73 # top in 2D and left, right, bottom, top, front, back in 3D). The value of a
74 # Dirichlet BC is specfied in the following way -- DIRCHILET: val -- where 'val' is
75 # the desired value. If the boundary condition is NON_UNIFORM_DIRICHLET, the
76 # boundary condition should be specified in the appropriate function in 'ICs_and_BCs.h'.
77 # Example 1: All periodic BCs for variable 'c'
78 # set Boundary condition for variable c = PERIODIC
79 # Example 2: Zero-derivative BCs on the left and right, Dirichlet BCs with value
80 # 1.5 on the top and bottom for variable 'n' in 2D
81 # set Boundary condition for variable n = NATURAL, NATURAL, DIRICHLET: 1.5, DIRICHLET: 1.5
82 
83 set Boundary condition for variable n = NATURAL
84 
85 # =================================================================================
86 # Set the model constants
87 # =================================================================================
88 # Set the user-defined model constants, which must have a counter-part given in
89 # customPDE.h. These are most often used in the residual equations in equations.h,
90 # but may also be used for initial conditions and nucleation calculations. The type
91 # options currently are DOUBLE, INT, BOOL, TENSOR, and [symmetry] ELASTIC CONSTANTS
92 # where [symmetry] is ISOTROPIC, TRANSVERSE, ORTHOTROPIC, or ANISOTROPIC.
93 
94 # The mobility, MnV in equations.h
95 set Model constant MnV = 1.0, DOUBLE
96 
97 # The gradient energy coefficient, KnV in equations.h
98 set Model constant KnV = 2.0, DOUBLE
99 ```
100 
101 The syntax for setting each input parameter is:
102 ```
103 set [parameter name] = [parameter value]
104 ```
105  The pound symbol (#) is used for comments, and any text after it is ignored by the file parser.

210 ## Note 1: Boundary Conditions
211 The boundary condition must be set for each variable, where each variable is given by its name, as defined in equations.h. The four boundary condition types are NATURAL, DIRICHLET, NON_UNIFORM_DIRICHLET and PERIODIC. If all of the boundaries have the same boundary condition, only one boundary condition type needs to be given. If multiple boundary condition types are needed, give a comma-separated list of the types. The order is the miniumum of x, maximum of x, minimum of y, maximum of y, minimum of z, maximum of z (i.e left, right, bottom, top in 2D and left, right, bottom, top, front, back in 3D). The value of a Dirichlet BC is specfied in the following way -- DIRCHILET: val -- where 'val' is the desired value. If the boundary condition is NON_UNIFORM_DIRICHLET, the boundary condition should be specified in the appropriate function in 'ICs_and_BCs.cc'. For vector variables, one boundary condition should be specified for each component.

229 ## Note 2: Checkpoint/Restart
230 The checkpoint/restart simulation allows one to continue from a previous simulation by loading the mesh and variable values from that previous simulation. One use of this system is to continue a simulation that stopped part of the way through, due to a hardware failure, running out of the allotted time on a cluster, etc. A second use is to use one simulation as the initial condition for another. Note that the simulated time carries over from the checkpoint. Thus if one ran a simulation to completion but wanted to see further evolution, one could load from the checkpoint created at the end of that simulation, but the desired number of time steps or simulation end time would have to be increased. Currently, the checkpoint system always read from files named ''restart.mesh'', ''restart.mesh.info'', and ''restart.time.info''. When a checkpoint is created, the previous checkpoint files have ''.old'' appended to their file names. To load from these older checkpoint files, the newer ones should be deleted (or moved) and the ''.old'' should be deleted from the names of the older files. An example of using the checkpoint/restart system can be seen in the 'dendriticSolidification' application.

232 ### Note 3: Loading Initial Conditions from File
233 Currently, initial conditions can only be read from vtk files (_not_ vtu files). Some variables can have initial conditions read from file and others can be specified in ICs_and_BCs.cc in the same application. For each variable, the user must set whether the file(s) to be read are in serial format, where all processors read from the same file, or parallel format, where all processors read from different files. If parallel format is selected, the desired domain decomposition must between the files and the simulation to be run. For non-adaptive meshes this will be true if the same number of cores is the same between the simulation that generated the vtk files and the simulation reading the vtk files. For adaptive meshes, obtaining the same domain decomposition may be difficult and merging parallel vtk files into a single file is likely the best approach. This process is planned to be cleaner in future versions of PRISMS-PF. An example of loading initial conditions from file can be seen in the 'allenCahn' application, using the 'parameters_pfield.in' input file.

235 ### Note 4: Nucleation
236 PRISMS-PF includes the capability for explicitly placing nuclei over the course of a simulation using an approach similar to the one described in the following publication:
237 
238 Jokisaari, Permann, and Thornton, A nucleation algorithm for the coupled conserved-nonconserved phase field model, \emph{Computational Materials Science}, 112, (2016).
239 
240 A nucleus of an non-conserved order parameter is placed within the computational domain determined by a probability given by the 'getNucleationProbability' function in the 'nucleation.cc' file. The variables that are allowed to nucleate are specified in the 'loadVariableAttributes' function in the 'equations.cc' file, as are the variable values needed to calculate the nucleation probability. The determination of when and where a nucleus should be seeded is determined in the core PRISMS-PF library, but the actual placement of the nucleus by modifying one of the model fields is expected to be performed in the 'explicitEquationRHS' function in the 'equations.cc' file. The placement of the nucleus is aided by the 'weightedDistanceFromNucleusCenter' function in the core library. The mobility for the nucleated field should be set to zero in the region surrounding the nucleus (the ''freeze zone'') for a specified amount of time (the ''freeze time''). Examples of nucleating particles can be found in the 'nucleationModel' and 'nucleationModel_preferential' applications.
241 
242 Currently, the nucleation parameters are one of the few places where the subsection' command is used in the input file. This command is used to set nucleation parameters separately for each nucleating variable. Each subsection is opened by ''subsection Nucleation parameters:'' followed by the variable name as set in 'equations.cc'. The end of each subsection block is concluded with the 'end' command. For example, if there are two nucleating order parameters **n1** and **n2**, the nucleation parameters section could look as follows:
243 ```
244 # =================================================================================
245 # Set the nucleation parameters
246 # =================================================================================
247 
248 set Time steps between nucleation attempts = 30
249 set Minimum allowed distance between nuclei = 50.0
250 set Order parameter cutoff value = 0.01
251 
252 subsection Nucleation parameters: n1
253  set Nucleus semiaxes (x, y, z) = 10, 5, 5
254  set Nucleus rotation in degrees (x, y, z) = 0, 0, 60
255  set Freeze zone semiaxes (x, y, z) = 15, 7.5, 7.5
256  set Freeze time following nucleation = 20
257  set Nucleation-free border thickness = 15
258 end
259 
260 subsection Nucleation parameters: n2
261  set Nucleus semiaxes (x, y, z) = 20, 10, 10
262  set Nucleus rotation in degrees (x, y, z) = 0, 0, 0
263  set Freeze zone semiaxes (x, y, z) = 30, 15, 15
264  set Freeze time following nucleation = 40
265  set Nucleation-free border thickness = 30
266 end
267 ```
268 The first three lines set the shared nucleation parameters. Next comes the subsection block for the variable **n1** and the subsection block for the variable **n2**.

270 ## Note 5: Model constants
271 Each application specifies its own set of model constants. These are most often used in the residual equations in the 'equations.cc' file, although they may also be used to specify initial conditions, non-uniform Dirichlet boundary conditions, nucleation probabilties, etc. Currently, five types of model constants are accepted: DOUBLE, INT, BOOL, TENSOR, and ELASTIC CONSTANTS. The use of these different types is as follows:
272 
273 - DOUBLE: For individual real numbers
274 - INT: For individual integers
275 - BOOL: For individual booleans (true/false)
276 - TENSOR: For either rank 1 tensors (i.e. vectors) or rank 2 tensors (i.e. matrices) with a size of the number of dimensions. These are assumed to be real numbers. Each row should be given by a comma-separated list in parentheses. For example, a 3D vector would be given as ```(2.5, 1.2, 0.1)``` and a 2D matrix would be given as: ```((4.1, 2.0),(1.6, 5.2))```.
277 - {[symmetry]} ELASTIC CONSTANTS: For sets of elastic constants, given as a vector. The symmetry options are ISOTROPIC, TRANSVERSE, ORTHOTROPIC, or ANISOTROPIC. The number of entries for the different symmetries are given below. PRISMS-PF converts these sets of elastic constants into a stiffness matrix.
278 
279 The number and order of the entries for the elastic constants are (where \f$C_{ij}\f$ are entries in the stiffness matrix):
280 
281 - ISOTROPIC (2D/3D): 2 constants (Young's Modulus, Poisson's Ratio)
282 - TRANSVERSE (3D): 5 constants (\f$C_{11}\f$, \f$C_{33}\f$, \f$C_{44}\f$, \f$C_{12}\f$, \f$C_{13}\f$)
283 - ORTHOTROPIC (3D): 9 constants (\f$C_{11}\f$, \f$C_{22}\f$, \f$C_{33}\f$, \f$C_{44}\f$, \f$C_{55}\f$, \f$C_{66}\f$, \f$C_{12}\f$, \f$C_{13}\f$, \f$C_{23}\f$)
284 - ANISOTROPIC (2D/3D): 21 constants (\f$C_{11}\f$, \f$C_{22}\f$, \f$C_{33}\f$, \f$C_{44}\f$, \f$C_{55}\f$, \f$C_{66}\f$, \f$C_{12}\f$, \f$C_{13}\f$, \f$C_{14}\f$, \f$C_{15}\f$, \f$C_{16}\f$, \f$C_{23}\f$, \f$C_{24}\f$, \f$C_{25}\f$, \f$C_{26}\f$, \f$C_{34}\f$, \f$C_{35}\f$, \f$C_{36}\f$, \f$C_{45}\f$, \f$C_{46}\f$, \f$C_{56}\f$)
285 
286 Each model constant in `parameters.in' must have a counterpart in the appropriate section of the `customPDE.h' file.
